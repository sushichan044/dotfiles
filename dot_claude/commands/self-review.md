# Self-Review Guide

## ADHDフレンドリー設計 + 障害防止強化版

### 📋 レビュー対象の背景

> このレビュー指示書は、開発者が作成したプルリクエストをレビューする際の
> チェック観点をまとめたものです。特に予期せぬ破壊的変更による障害リスクの見落としを防ぐことを重視しています。

---

## 🚨 公開API削除影響チェック

### 🔴 STEP 1: 公開API削除の自動検出

まず、ブランチ分岐元からの差分を取り、見るべきファイルを見極めます。

`ANCESTOR_COMMIT=$(git merge-base HEAD $(git remote show origin | grep 'HEAD branch' | awk '{print $NF}')); git diff $ANCESTOR_COMMIT --name-only`

### 🚨 STEP 2: 削除された公開API一覧作成（必須）

**🔴 チェック必須項目 - 絶対に見逃すな！**

- [ ] 🚨 **削除された public functions/methods** を全てリストアップ
- [ ] 🚨 **削除された public classes/interfaces** を全てリストアップ
- [ ] 🚨 **削除された exported constants/variables** を全てリストアップ
- [ ] 🚨 **削除された API endpoints/URL paths** を全てリストアップ
- [ ] 🚨 **削除された database tables/columns** を全てリストアップ
- [ ] 🚨 **削除された configuration items** を全てリストアップ

### 🔴 STEP 3: 影響範囲調査（絶対実行）

**各削除項目について必ず実行:**

```bash
# 全プロジェクトで参照箇所を検索
rg "削除されたAPI名" --type-add 'source:*.{js,ts,jsx,tsx,py,java,go,rs,php,rb,cs}' -t source

# 設定ファイルでの参照確認
rg "削除されたAPI名" --type-add 'config:*.{json,yml,yaml,toml,ini,env}' -t config
```

**🚨 影響調査必須チェックリスト:**

- [ ] 🚨 **他のファイルからの直接参照**を全て確認
- [ ] 🚨 **テストファイルでの使用**を全て確認
- [ ] 🚨 **設定ファイルでの参照**を全て確認
- [ ] 🚨 **ドキュメントでの言及**を全て確認
- [ ] 🚨 **外部システムからの呼び出し**を全て確認

### 🟡 STEP 4: 障害リスク評価

**🚨 各削除項目の障害リスク評価（必須）:**

- [ ] 🔴 **CRITICAL**: プロダクション即時停止リスク
- [ ] 🟡 **HIGH**: ビジネスクリティカル機能影響リスク
- [ ] 🟢 **MEDIUM**: サポート機能影響リスク
- [ ] ⚪ **LOW**: 将来的影響リスク

### 🟢 STEP 5: 段階的検証（必須実行）

**🚨 環境別動作確認（全て実行）:**

- [ ] 🚨 **開発環境**での動作確認完了
- [ ] 🚨 **テスト環境**での動作確認完了
- [ ] 🚨 **ステージング環境**での動作確認完了
- [ ] 🚨 **統合テスト**の実行・成功確認完了
- [ ] 🚨 **リグレッションテスト**の実行・成功確認完了

### ✅ 公開API削除影響チェック完了確認

**レビュー時に以下が適切に実施されているか確認:**

- [ ] STEP 1: 公開API削除の検出が実行されている
- [ ] STEP 2: 削除された公開API一覧が作成されている
- [ ] STEP 3: 影響範囲調査が実施されている
- [ ] STEP 4: 障害リスク評価が行われている
- [ ] STEP 5: 段階的検証が実行されている

## 🔐 セキュリティレビューマトリックス

### OWASP Top 10 チェック

- [ ] **A01 - Broken Access Control**: 認証・認可の実装は適切か
- [ ] **A02 - Cryptographic Failures**: 暗号化・ハッシュ化は適切か
- [ ] **A03 - Injection**: SQLインジェクション等の脆弱性はないか
- [ ] **A04 - Insecure Design**: セキュアな設計パターンが使われているか
- [ ] **A05 - Security Misconfiguration**: セキュリティ設定は適切か
- [ ] **A06 - Vulnerable Components**: 脆弱性のある依存関係はないか
- [ ] **A07 - Identification Failures**: 認証機能に問題はないか
- [ ] **A08 - Software Integrity Failures**: ソフトウェア整合性は保たれているか
- [ ] **A09 - Security Logging Failures**: セキュリティログは適切か
- [ ] **A10 - Server-Side Request Forgery**: SSRF脆弱性はないか

### データ保護チェック

- [ ] 個人情報・機密データの適切な処理
- [ ] データの暗号化（保存時・転送時）
- [ ] ログにセンシティブ情報が含まれていないか
- [ ] API キー・トークンがハードコードされていないか

## ⚡ パフォーマンスレビューマトリックス

### 計算量・効率性

- [ ] **時間計算量**: O(n), O(n²), O(log n) の適切性
- [ ] **空間計算量**: メモリ使用量の最適化
- [ ] **ループ効率**: 不要な反復処理の排除
- [ ] **データ構造**: 用途に適したデータ構造の選択

### リソース使用量

- [ ] メモリリークの可能性
- [ ] CPU使用率の影響
- [ ] I/O操作の効率性
- [ ] ネットワーク帯域の考慮

### データベース・API

- [ ] **N+1問題**: クエリの効率性
- [ ] **インデックス**: 適切なインデックス設計
- [ ] **キャッシュ**: 適切なキャッシュ戦略
- [ ] **ページネーション**: 大量データの適切な処理
